# 🪻 設計ドキュメント・プロンプト集（Word出力用｜うさうさ研修工房）

**用途**：アップロードした「単一HTMLツール」のソースから、文書種別ごとに **Wordファイル(.docx)** を作らせる再利用プロンプト集。
**使い方**：(1) まず下の「共通プリアンブル」をコピー → (2) 作りたい文書の「本体プロンプト」をその後ろに貼る → (3) 対象HTMLと一緒にAIへ渡す。先頭の `{{…}}` を埋める。
**ラインナップ**：①要件定義書 ②詳細設計書 ③実装ガイド（実装しよう）④アーキテクチャ仕様書 ⑤データフロー仕様書 ＋ボーナス（⑥テスト仕様書 ⑦利用ガイド/README）。

> 方針：5種を**別ファイルに分ける**ことで、読者と目的を分離します（要件＝何を/なぜ、設計＝どう作る、実装＝手を動かす順、アーキ＝全体の骨格、データフロー＝データの流れ）。

---

## 変数（共通・埋める）
| 変数 | 例 |
|---|---|
| `{{TOOL_NAME}}` | ラベンダー研修スライド・ジェネレーター（アーティファクト版） |
| `{{BRAND}}` | うさうさ研修工房 |
| `{{VERSION}}` | v1.0.0 PRO |
| `{{DATE}}` | 2026-06-13 |
| `{{SOURCE_FILENAME}}` | lavender_slide_studio_app_v1_0_0_pro_20260613.html |
| `{{SPECIAL_OUTPUT}}` | .pptx(OOXML/ZIP) ／ なければ「なし」 |
| `{{AUDIENCE}}` | （文書ごとに本体側で指定） |

## ★ 共通プリアンブル（各本体プロンプトの先頭に必ず貼る）

```markdown
# 共通ルール（最優先・違反は重大）
あなたは商用ソフトの設計文書を書く一流のテクニカルライター兼設計者です。日本語で執筆します。
対象は単一HTMLツール「{{TOOL_NAME}}」（{{BRAND}} ／ {{VERSION}} ／ {{DATE}}）。ソースは添付の {{SOURCE_FILENAME}}（無ければ会話中のソース）。

1. ウソなく：色・座標・サイズ・HTTPヘッダ・モデル名・定数・関数名・既定値・プロンプト等は、ソースから抽出した実値のみ。記憶・一般論・推測で埋めない。
2. 未確認は明記：ソースに無い／読めない項目は「（ソースに記載なし・未確認）」と書く。創作で補わない。
3. 事実と所感の分離：客観仕様は通常記述、主観・設計判断の背景は【所感】ラベルで区別。
4. 実名を出さない：人物は役職名のみ（メイン講師／サブ講師／上長／人事担当 等）。個人名・メール・APIキー・トークンは記載しない。
5. 効果指標を捏造しない：時短率・満足度などは実測の引用が無い限り書かない。
6. たとえ話（例：うさうさラーメン店）を使う場合は「※このたとえは正確ではありません」と必ず明示。
7. まず添付ソースを精読して実値を抽出してから書き始める。読めない箇所は創作せず「未確認」として列挙。

# Word(.docx)出力指定（必須）
- 成果物は **Wordファイル(.docx)** として作成し、ダウンロードできる形で提示する（本文をチャットに貼るだけにしない）。
- ページ：US Letter 縦（または A4 縦）、余白1インチ。本文フォント「Yu Gothic」10.5pt。
- 見出し：H1/H2/H3 を使用（色 #4A3370 / #4A3370 / #7558B0、Yu Gothic 太字）。先頭は H1 のタイトル。
- 表：枠線つき・列幅は本文幅に収める・ヘッダ行に淡い背景（#E6DEF5）。仕様マトリクスは表で。
- コード／擬似コード／ファイル一覧／逐語プロンプトは等幅（Consolas）かつ淡い網掛け（#F3EFFA）。日本語を含むコードは可読性優先で可。
- フッターにページ番号と「{{BRAND}} ／ {{DATE}}」。ファイル名は ASCII＋日付（例：toolname_doctype_v1_0_0_20260613.docx）。
- 箇条書きは要点に限り簡潔に。過剰な太字・絵文字の乱用はしない。
- ハード制約（単一HTML・外部依存ゼロ・ES5互換・全innerHTMLのエスケープ・[hidden]{display:none!important}・ASCII日付ファイル名・バージョン刻印5か所一致・実名なし）は、ソースで満たされているか検証し結果を書く（満たさない点は正直に指摘）。
- 末尾に【付記】：出典・査読前の初稿である旨・実名なし運用。任意で高杉晋作「面白きこともなき世を面白く」を添える。
```

## ① 要件定義書（Requirements Definition）
*読者：発注/承認者・メイン講師・人事担当。「何を・なぜ」を中心に。*

```markdown
（↑共通プリアンブルを先頭に貼る）

# この文書：要件定義書
読者は {{AUDIENCE:発注・承認者／メイン講師／人事担当}}。実装詳細より「目的・スコープ・要求」を中心に書く。

# 出力構成（H1タイトル＋メタ情報表のあとに）
1. 背景と目的（なぜ作るか。解決したい課題。ゴール）
2. スコープ（対象／対象外を明確に）
3. ステークホルダーと役割（役職名のみ）
4. 利用シナリオ／ユースケース（主要フローを箇条書きまたは表）
5. 機能要件 FR（表：ID／要求／優先度／受け入れ基準）※実装根拠を脚注的に併記
6. 非機能要件 NFR（表：性能・セキュリティ・互換性・アクセシビリティ・保守性・可搬性）
7. 制約・前提（ハード制約。動作環境。外部依存の有無）
8. 用語定義
9. 受け入れ条件（完了の定義／テスト観点の要約）
10. リスクと未決事項（未確認は「未確認」と明記）
11. 付記（事実と所感／出典／査読前初稿／実名なし）

# 注意
- FR/NFR は本文HTMLの実機能から起こす（存在しない要求を足さない）。
- 受け入れ条件は検証可能な形（観測できる事実）で書く。
```

## ② 詳細設計書（Detailed Design）
*読者：実装/保守担当。「どう作るか」を実装根拠で。*

```markdown
（↑共通プリアンブルを先頭に貼る）

# この文書：詳細設計書（実装根拠版）
読者は {{AUDIENCE:実装・保守担当}}。この1枚で同一実装を再現できる粒度。

# 出力構成
1. 概要と設計方針（ハード制約・実装で遵守を確認したもの）
2. データ構造（スキーマをコードブロック＋不変条件・異常系の扱い）
3. デザイン仕様（CSS変数の実値表・色マップ・図形/SVGの実数値・フォント・影）
4. 画面構成（DOMツリーをコードブロック）＋フォーム既定値（実option値・checked）
5. 関数仕様（表：関数／引数→戻り値→副作用。逐語に近い精度で全関数）
6. 主要処理フロー（擬似コード）
7. 外部連携／API（実値表：URL・送信ヘッダ・メソッド・ボディ[model/max_tokens]・解析・エラー処理）
8. 重要文字列の全文転記（プロンプト等：逐語・改変禁止）
9. 特殊出力生成仕様（{{SPECIAL_OUTPUT}}：同梱ファイル一覧／座標・単位[EMU等]／署名値／検証要件。無ければ該当なし）
10. 書き出し・印刷仕様
11. バージョン刻印（実在箇所と実値・一致の明記）
12. エラー処理・ログ設計
13. 既知の制限・非対応
14. 付記（事実と所感／出典／査読前初稿／実名なし）
```

## ③ 実装ガイド「実装しよう」（Implementation Guide）
*読者：これから手を動かす人。順番・落とし穴・コード断片中心。難易度信号つき。*

```markdown
（↑共通プリアンブルを先頭に貼る）

# この文書：実装ガイド「実装しよう」
読者は {{AUDIENCE:新任エンジニア／写経で学ぶ人}}。ゼロから組む順番と、つまずきの回避を実装根拠で示す。各手順に難易度信号 🔴(難)/🟡(中)/🟢(易) を付ける。

# 出力構成
1. はじめに（完成物の姿・前提・必要なもの）
2. 全体像（何をどの順で作るか。ロードマップ）
3. ステップ・バイ・ステップ（各ステップ：目的→やること→**実コードに基づく要点コード断片**→確認方法→🔴🟡🟢）
   - 例：土台HTML/CSS設計トークン → ユーティリティ(esc/arr/escape徹底) → フォーム&既定値 → 生成フロー(API) → 描画 → 書き出し(MD/メモ/印刷/特殊出力) → ログ → 仕上げ(刻印)
4. ハード制約チェックリスト（実装中に守る項目）
5. つまずきポイント（実際にソースで対策されている防御＝事実として。例：型強制・配列ガード等）
6. 動作確認・QAの回し方（テスト観点・既知値）
7. 仕上げ（バージョン刻印5か所の合わせ方）
8. 付記（事実と所感／出典／査読前初稿／実名なし）

# 注意
- コード断片は本文HTMLの実装から抜く（疑似・改変は「概略」と明記）。
- 「人ではなく仕組み」で守る観点（防御的実装）を、事実ベースで強調してよい。
```

## ④ アーキテクチャ仕様書（Architecture Specification）
*読者：レビュア・設計の意思決定者。骨格・責務・原則・拡張点。*

```markdown
（↑共通プリアンブルを先頭に貼る）

# この文書：アーキテクチャ仕様書
読者は {{AUDIENCE:レビュア／設計意思決定者}}。全体の骨格と「なぜこの形か」を実装根拠＋【所感】で。

# 出力構成
1. システムコンテキスト（このツールと外部[ブラウザ／生成AIエンドポイント]の関係。テキストの構成図可）
2. 全体アーキテクチャ（単一HTML内のレイヤ／モジュール分割：UI・状態・生成・解析・描画・書き出し・ログ）
3. コンポーネント責務と依存関係（表：モジュール／責務／依存先。テキストの依存図）
4. 設計原則・制約（外部依存ゼロ／ES5互換／単一ファイル／オフライン可／キー非保持 などの根拠）
5. 横断的関心事（セキュリティ＝全innerHTMLのエスケープ、ロギング、エラー処理、国際化/日本語、アクセシビリティ）
6. データ所有と状態（どこが真実の源か：currentDeck 等の状態と寿命）
7. 外部インターフェース方針（APIの呼び方・送信ヘッダ方針・認証の扱い）
8. 特殊出力の生成方式（{{SPECIAL_OUTPUT}}：ライブラリ非依存で組む方式の概要と根拠。無ければ該当なし）
9. 拡張・変更の指針（足す/替えるときの安全な勘所）
10. 設計判断の記録（ADR風：決定／背景／代替案／結果。【所感】明示）
11. 既知の制限・トレードオフ
12. 付記（事実と所感／出典／査読前初稿／実名なし）

# 注意
- 図はテキスト（コードブロックの箱と矢印）で表現してよい。実在しない層を作らない。
```

## ⑤ データフロー仕様書（Data Flow Specification）
*読者：処理の流れ・検証点・信頼境界を追う人。*

```markdown
（↑共通プリアンブルを先頭に貼る）

# この文書：データフロー仕様書
読者は {{AUDIENCE:実装・レビュア}}。入力から出力までのデータの流れ・変換・検証点を実装根拠で。

# 出力構成
1. 目的と範囲
2. データ・ディクショナリ（主要データの定義：入力フィールド／中間データ／slideスキーマ等。表）
3. コンテキストDFD（Level 0：ツールと外部[利用者／生成AI]の入出力。テキスト図）
4. 詳細DFD（Level 1：フォーム入力→プロンプト生成→API送信→レスポンス→JSON解析→デック→描画／各書き出し[MD・メモ・印刷・特殊出力]。テキスト図＋各変換の説明）
5. 各変換ステップの仕様（入力→処理→出力／実関数名／実値）
6. バリデーション・エスケープ点（どこで必須入力チェック／エスケープ／配列・型ガードが入るか＝事実）
7. 状態と永続化（メモリ状態・localStorage等の使用有無＝事実。キー非保持等）
8. 信頼境界とセキュリティ（外部から来るデータ＝AI応答の扱い、innerHTML前のエスケープ、認証情報の非保持）
9. エラー時のデータ経路（失敗時に何がどう表示/記録されるか）
10. 付記（事実と所感／出典／査読前初稿／実名なし）

# 注意
- 「どこでエスケープ/検証されるか」はソースの実装位置に基づく事実として書く。
```

## 運用Tips
- **5本まとめて作りたい**とき：共通プリアンブル＋各本体を順に渡し、「①〜⑤を**それぞれ別の.docx**として出力」と指示。1ファイルに混ぜない（読者と目的が違うため）。
- **Notionにも置く**とき：各本体末尾に「同内容を、指定Notion親ページ配下の子ページにも作成（H1はタイトルへ分離・日本語UTF-8直書き・アイコン絵文字は付けない）」を追記。
- **記事化**したいとき：要件/設計ではなく「読者にやさしい技術ライター」役へ替え、WHY→WHAT→HOW＋🔴🟡🟢＋🍜（不正確と明記）で。

【事実】本書は文書種別ごとにWord(.docx)を生成させるためのプロンプト集。共通プリアンブルで「実値のみ・未確認明記・実名なし・事実と所感の分離」を全文書に効かせています。
【所感】文書を読者別に割ると、1枚あたりが軽くなり、直しも速くなります。「全部入り1枚」より「目的別の薄い束」のほうが現場で生きます。
