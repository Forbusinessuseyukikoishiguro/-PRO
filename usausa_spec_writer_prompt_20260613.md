# 🪻 設計書ジェネレーター・プロンプト（うさうさ研修工房）
**用途**：アップロードした「単一HTMLツール」のソースを読み、**実装そのものを根拠にした**要件定義・詳細実装設計書をMarkdownで出力させる、再利用メタプロンプト。
**使い方**：下の「── プロンプト本体 ──」以降を丸ごとコピーし、対象のHTML（または貼り付けソース）と一緒にAIへ渡す。先頭の `{{…}}` を埋める。

> 設計方針：このプロンプトは「ウソなく・実値のみ・未確認は明記」を最優先にしています。AIが記憶や推測で値を埋めることを禁じ、ソースから抽出させます。

---

## 変数（埋める）
| 変数 | 例 |
|---|---|
| `{{TOOL_NAME}}` | ラベンダー研修スライド・ジェネレーター（アーティファクト版） |
| `{{BRAND}}` | うさうさ研修工房 |
| `{{VERSION}}` | v1.0.0 PRO |
| `{{DATE}}` | 2026-06-13 |
| `{{SOURCE_FILENAME}}` | lavender_slide_studio_app_v1_0_0_pro_20260613.html |
| `{{SPECIAL_OUTPUT}}` | .pptx(OOXML/ZIP) ／ なければ「なし」 |
| `{{AUDIENCE}}` | 社内講師・新任エンジニア |

---

── プロンプト本体（ここからコピー）──

```markdown
# 役割
あなたは、商用ソフトウェアの**要件定義書・詳細設計書を書く一流のテクニカルライター兼ソフトウェア設計者**です。日本語で、保守担当者がこの1枚だけを見て同一実装をゼロから再現できる粒度の設計書を作成します。

# 入力
- 対象：単一HTMLツール「{{TOOL_NAME}}」（{{BRAND}} ／ {{VERSION}} ／ {{DATE}}）
- ソース：添付の {{SOURCE_FILENAME}}（添付がなければ会話中に貼られたソース）。**必ず実ファイルを読み、実値を抽出すること。**
- 特殊な出力生成：{{SPECIAL_OUTPUT}}
- 想定読者：{{AUDIENCE}}

# 最優先の原則（違反は重大）
1. **ウソなく**：色・座標・サイズ・HTTPヘッダ・モデル名・定数・関数名・既定値・プロンプト文字列などは、**ソースから抽出した実値のみ**を書く。記憶・一般論・推測で埋めない。
2. **未確認は明記**：ソースに無い／読めない項目は「（ソースに記載なし・未確認）」と書く。創作で補わない。
3. **事実と所感の分離**：客観仕様は通常記述、設計判断の背景・主観は **【所感】** ラベルを付けて区別する。
4. **実名を出さない**：人物は役職名のみ（メイン講師／サブ講師／上長／人事担当 等）。個人名・メール・鍵・トークンは記載しない（APIキーは「保持しない／入力欄なし」等の事実のみ）。
5. **効果指標を捏造しない**：時短率・満足度などの数値は、実測の引用が無い限り書かない。

# 抽出タスク（ソースから必ず拾う）
- バージョン刻印が**何か所**にあり一致しているか（ファイル名／HTMLコメント／CHANGELOG／画面バッジ／フッター／起動ログ 等）。
- CSSの設計トークン（`:root` のCSS変数：名前・値・用途）、フォント、影。
- 主要定数（色マップ・列挙・既定値）と、**全関数のシグネチャと実装挙動**（引数→戻り値→副作用）。
- フォーム/UIの**既定値**（select の option、checkbox の checked、value 形式）。
- 外部連携：エンドポイントURL・**送信ヘッダ**・メソッド・ボディ（model 名・max_tokens 等）・レスポンス解析・エラー処理。
- AIや外部に渡す**重要文字列（プロンプト等）は全文を逐語転記**（改変しない）。
- 特殊フォーマット生成（例：OOXML/ZIP、PDF印刷CSS）の**ファイル構成一覧・座標・単位・署名値**など。
- 入力データ構造（スキーマ）と不変条件、異常系の扱い。
- 既知の制限・非対応。

# 出力（Markdown・この構成と順序で）
冒頭に H1 のタイトル（「{{TOOL_NAME}} 要件定義・詳細実装設計書（実装根拠版） {{VERSION}}」）。続いてメタ情報表。以降：
1. 概要と目的（＋ハード制約。実装で遵守を確認したものに限る）
2. 用語定義（表）
3. 機能要件 FR（表：ID／機能／**実装関数**）
4. 非機能要件 NFR（表：ID／区分／**実装での担保**）
5. データ構造（スキーマをコードブロック＋不変条件）
6. デザイン仕様（CSS変数の実値表・色マップ・図形/SVGの実数値）
7. 画面構成（DOMツリーをコードブロック）＋フォーム既定値（実option値）
8. 関数仕様（表：関数／実装。逐語に近い精度で）
9. 主要処理フロー（擬似コードをコードブロック）
10. 外部連携／API（実値表）
11. 重要文字列の全文転記（プロンプト等：コードブロックで逐語）
12. 特殊出力生成仕様（{{SPECIAL_OUTPUT}}：同梱ファイル一覧／座標・単位／署名値／検証要件。無ければ「該当なし」）
13. 書き出し・印刷仕様
14. バージョン刻印（**実際に存在した箇所と実値**を列挙し、一致を明記）
15. 品質保証（テスト設計：区分・観点・既知値・**実際に修正した不具合**があれば事実として記載）
16. 見本データ／初期データ（あれば）
17. 再現・QAワークフロー（コードブロックの手順）
18. 変更履歴（CHANGELOG：ソース記載を転記）
19. 付記（事実と所感／出典／査読前初稿である旨／実名なし）

# 体裁ルール
- 表は仕様マトリクスに、コードブロックはコード・擬似コード・ファイル一覧・逐語プロンプトに使う。
- 箇条書きは要点に限り簡潔に。過剰な太字や絵文字の乱用はしない。
- たとえ話（例：🍜うさうさラーメン店）を使う場合は「※このたとえは正確ではありません」と明示する。
- 文末に高杉晋作の句「面白きこともなき世を面白く」を任意で添えてよい（社内文書のトーン）。
- ASCIIファイル名・日付スタンプ運用、単一HTML・外部依存ゼロ・ES5互換・全innerHTMLのエスケープ・`[hidden]{display:none!important}`・実名なし、といった**ハード制約はソースで満たされているか検証し、結果を書く**（満たさない点は正直に指摘）。

# 仕上げ前のセルフチェック（満たすまで直す）
- [ ] すべての数値・文字列は**ソース由来**か。記憶/推測で書いた箇所はないか。
- [ ] 未確認項目に「（未確認）」を付けたか。
- [ ] 【所感】と事実を混在させていないか。
- [ ] 個人名・鍵・トークンを書いていないか。
- [ ] 重要プロンプト等を**逐語**転記したか（要約していないか）。
- [ ] この設計書だけで再現できる粒度か（欠けたら該当章を補う）。

まず添付ソースを読み、抽出が終わってから本文を書き始めてください。読めない箇所があれば、創作せず「未確認」として列挙してください。
```

## 補足（運用Tips）
- **記事を書かせたい**ときは、本体冒頭の「役割」を「読者にやさしく解説する技術ライター」に替え、構成を **WHY→WHAT→HOW**、難易度信号 **🔴🟡🟢**、🍜ラーメン店アナロジー（不正確と明記）に差し替えると、ALJ/うさうさ先生トーンの社内記事プロンプトになります。
- **Word/Notion 納品**まで含めたいときは、本体末尾に一行追加：「完成後、同内容をWord(.docx)と、指定のNotion親ページ配下の子ページにも出力する（H1はタイトルへ分離、日本語はUTF-8直書き、アイコン絵文字は付けない）。」
- **査読フロー**：出力には常に「査読前の初稿」と明記させ、公開前にメイン講師・人事担当の確認を通す運用に。

【事実】本プロンプトは、実装根拠版の設計書を生成するための指示テンプレートです。
【所感】「実値しか書かせない・未確認は正直に出させる」を芯にすると、設計書はだいぶ信頼できる道具になります。
